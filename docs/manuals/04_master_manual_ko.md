# NEWMSO 마스터용 기능 설명서

## 1. 문서 목적

이 문서는 최고 권한 운영자용 설명서입니다.  
현재 원본 코드 기준으로 마스터 전용 별도 메뉴가 따로 존재하는 구조는 아니며, 마스터는 `MSO 권한 기반 관리자 화면 전체`를 최고 권한 범위로 운영하는 사용자입니다.

## 2. 마스터의 역할

마스터는 단순 관리자가 아니라 운영 기준을 최종 결정하는 책임자입니다.

| 역할 | 설명 |
| --- | --- |
| 최고 권한 관리 | 회사/권한/데이터 관리 최종 통제 |
| 정책 승인 | 위험 작업의 승인 여부 판단 |
| 감사 대응 | 접근기록과 변경기록 점검 |
| 장애 대응 | 운영 이상 시 최종 판단 |
| 배포 기준 관리 | 무엇을 언제 운영에 반영할지 결정 |

## 3. 마스터가 실제로 사용하는 화면

현재 원본 구조에서 마스터는 별도 UI가 아니라 아래를 모두 사용합니다.

- 관리자 메뉴 전체
- 인사관리 전체
- 전자결재 전체
- 재고관리 전체
- 내정보 관리자형 대시보드

즉, 마스터는 `관리자 설명서의 상위 운영자`라고 이해하면 됩니다.

## 4. 마스터가 중점적으로 통제해야 할 화면

### 회사관리

- 회사 구조와 운영 틀을 관리합니다.
- 다회사 운영 기준이 있다면 핵심 화면입니다.

### 직원권한

- 메뉴, 기능, 관리자 범위를 최종 통제합니다.
- 권한 오배포는 보안사고로 이어질 수 있습니다.

### 접근감사로그 / 감사로그

- 누가 접근했고 무엇을 바꿨는지 추적합니다.
- 운영 사고 분석의 기준입니다.

### 데이터백업

- 복구 가능한 시점을 확보합니다.
- 구조 변경 전 반드시 확인해야 합니다.

### 데이터초기화

- 가장 위험한 기능 중 하나입니다.
- 단독 실행 금지에 가까운 통제 기준이 필요합니다.

### 양식빌더 / 문서서식

- 전사 결재 문서와 출력 형식을 바꾸는 기능입니다.
- 한 번 잘못 바꾸면 많은 화면이 동시에 영향받습니다.

### 팝업관리

- 외부 사이트에 노출될 수 있는 데이터를 관리합니다.
- 대외 공지 성격으로 다뤄야 합니다.

## 5. 마스터 운영 원칙

### 최소 권한 원칙

- 모든 직원에게 필요한 메뉴만 열어야 합니다.
- 편의상 넓게 주는 방식은 장기적으로 위험합니다.

### 변경 기록 원칙

- 권한 변경, 데이터 보정, 문서 구조 변경은 이유를 남겨야 합니다.
- 구두 요청만으로 바로 반영하지 않는 것이 안전합니다.

### 작업 전 백업 원칙

- 큰 변경 전에 백업을 먼저 확보해야 합니다.
- 복구 기준 시점을 항상 명확히 해야 합니다.

### 운영/테스트 분리 원칙

- 운영 계정에서 실험성 변경을 하지 않는 것이 좋습니다.
- 검증 후 반영 흐름을 유지해야 합니다.

## 6. 마스터 일일 점검 항목

1. 긴급 장애나 권한 사고 제보 확인
2. 결재 적체 여부 확인
3. 재고 부족, 운영 알림, 공지 상태 확인
4. 접근 이상 징후 확인
5. 관리자 문의 건 확인

## 7. 마스터 주간 점검 항목

1. 권한 변경 내역 검토
2. 감사로그와 접근감사로그 검토
3. 백업 상태 점검
4. 전자결재 양식 변경 여부 검토
5. 문서서식 변경 여부 검토
6. 팝업 노출 상태 점검

## 8. 마스터 월간 점검 항목

1. 회사별 메뉴 노출 샘플 점검
2. 관리자 계정 목록 정리
3. 오래된 권한과 계정 정리
4. 인사, 급여, 재고, 결재 핵심 데이터 이상치 검토
5. 고위험 기능 사용 이력 검토

## 9. 마스터가 자주 하는 업무 흐름

### 권한 체계 정비

1. 직원권한에서 현재 상태 확인
2. 부서별 표준 권한 정의
3. 예외 권한 최소화
4. 샘플 계정으로 실제 메뉴 확인

### 운영 사고 대응

1. 사용자 제보 확인
2. 감사로그와 접근감사로그 확인
3. 권한 문제인지 데이터 문제인지 구분
4. 필요 시 백업 기준 시점 검토

### 양식/문서 구조 변경 승인

1. 변경 목적 확인
2. 영향 범위 확인
3. 실제 사용 부서 확인
4. 백업 후 반영

### 초기화 또는 대량 수정 승인

1. 작업 필요성 검토
2. 백업 여부 확인
3. 영향 범위 문서화
4. 실행자와 복구 계획 확정 후 진행

## 10. 현재 원본 구조에서 마스터가 특히 알아야 할 점

- 관리자 메뉴는 사실상 MSO 중심 구조입니다.
- 조직도는 권한 체계에 존재하지만 현재 기본 사이드바에 고정 메뉴로 보이지 않습니다.
- 전자결재 관리 탭은 현재 일반 사용자에게도 보일 수 있으므로, 운영 교육 시 사용 대상 구분을 명확히 해줘야 합니다.
- 일부 설계 의도와 실제 화면 노출이 완전히 같지 않으므로, 운영 기준 문서는 코드와 별도로 유지할 필요가 있습니다.

## 11. 최종 정리

현재 구조에서 마스터는 “더 많은 화면을 보는 사람”이라기보다, “운영 기준을 정하고 위험 작업을 통제하는 사람”입니다.  
따라서 마스터 운영의 핵심은 기능 사용 자체보다 `무엇을, 언제, 어떤 절차로 바꿀 것인지`를 관리하는 데 있습니다.
