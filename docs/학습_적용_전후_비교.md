# 자료 학습·적용 전 vs 후 비교 (장단점)

“자료를 주고 학습해서 적용해달라”고 하기 **전**과 **후**의 차이를 정리한 문서입니다.

---

## 1. 전(Before) — 자료 없이 연계 점검만 했을 때

### 했던 일
- **인사관리 전체 연계 점검** 요청에 따라, 코드/DB 기준으로만 검토
- **연차**: 전자결재·전자결재_개선에서 `annual_leave` 사용 → `annual_leave_used` 통일
- **근태**: 마이페이지 출퇴근기록이 `attendance`만 기록 → `attendances` 동기화 추가
- **전자결재 연차 승인**: 휴가 기간 전체에 대해 `attendance`/`attendances` 일별 반영

### 장점
- **기존 코드·스키마에만 맞춤** → 이미 있는 구조를 깨지 않고 일관성만 맞출 수 있음
- **변경 범위가 명확** → “연차 컬럼 불일치”, “근태 테이블 이원화”처럼 이슈 단위로 수정 가능
- **참고 제품 없이도** DB/API 연동 오류는 해소 가능
- **구현 속도**가 상대적으로 빠름 (화면/업무 규칙 설계 없이 수정 위주)

### 단점
- **업무·화면 기준이 없음** → “실제로 급여·계약을 이렇게 쓰고 싶다”는 목표가 코드 역추적에만 의존
- **누락 가능성** → 참고 시스템에 있는 기능(선지급, 당직수당, 공제 상세, 근로·임금 정보 등)을 놓치기 쉬움
- **문서화 부족** → 나중에 “왜 이렇게 했지?”를 설명할 스펙/정책 문서가 없음
- **디자인·UX**는 기존 화면 그대로라, 참고하고 싶은 UI와 차이가 남

---

## 2. 후(After) — 자료 주고 학습·적용한 뒤

### 했던 일
- **참고 화면(급여·비과세·과세·공제·근무일·명세서·계약·급여설정 등)** 을 정리해 `docs/급여시스템_학습정리.md` 작성
- **당직수당(야간)** 비과세 항목을 DB·급여정산·구성원·명세서에 반영
- **4대보험 계산 기준**(과세표준 = 급여−비과세)을 코드 주석으로 명시
- **선지급·공제 상세·근로·임금 정보·계약서 종류·급여설정** 등은 “적용 시 권장 사항”으로 정리 (추후 단계별 적용용)

### 장점
- **목표가 분명해짐** → “이런 화면/이런 규칙으로 맞추고 싶다”가 문서로 남아, 개발·기획이 같은 기준을 봄
- **누락 감소** → 참고 자료에 있는 항목(비과세 종류, 공제 항목, 계약서 종류, 급여설정 항목 등)을 체크리스트처럼 활용 가능
- **단계별 적용 가능** → 한 번에 다 구현하지 않고, 학습 문서 보고 우선순위 정해서 조금씩 반영 가능
- **유지보수·온보딩**에 유리 → 새로 합류한 사람이 “급여/계약은 이렇게 쓰는 시스템”이라는 걸 문서로 이해 가능
- **법·제도 반영** → 근로기준법, 개인정보보호법, 국고금관리법 등 참고 자료에 나온 조항을 문서에 적어 두고 나중에 코드에 반영하기 쉬움

### 단점
- **참고 자료에 종속** → 참고 제품(메디플로우 등)과 다른 정책을 쓰고 싶을 때, 문서를 다시 정리해야 함
- **초기 비용** → 자료 이해·문서 작성·일부 적용까지 시간이 더 듦
- **미적용 항목 많음** → 선지급, 공제 상세, 근로·임금 정보 뷰, 계약서 작성 UI 등은 아직 “스펙만 정리” 상태라, 실제 구현은 추가 작업 필요
- **자료 품질 의존** → 스크린샷/설명이 부정확하면 문서와 실제 요구가 어긋날 수 있음

---

## 3. 요약 비교표

| 구분 | 전 (자료 없이 연계 점검) | 후 (자료 학습·적용) |
|------|--------------------------|----------------------|
| **기준** | 기존 코드·DB 구조 | 참고 화면·업무 규칙 + 기존 코드 |
| **변경 특성** | 연동 버그/불일치 수정 위주 | 스펙 문서화 + 필요한 부분만 단계적 반영 |
| **문서** | 거의 없음 | `급여시스템_학습정리.md` 등 정리 문서 있음 |
| **장점** | 빠르게 연계만 맞출 수 있음, 기존 구조 유지 | 목표 명확, 누락 줄고, 단계별·유지보수 용이 |
| **단점** | 업무/화면 기준 부재, 기능 누락·문서 부족 | 참고에 의존, 초기 비용·미구현 항목 많음 |

---

## 4. 어떤 방식을 쓸지

- **당장 연동만 맞추고 싶을 때**  
  → 전처럼 “코드/DB 기준 연계 점검”만 해도 됨. (이미 연차·근태는 그렇게 수정됨.)

- **급여·계약을 참고 제품처럼 체계적으로 가져가고 싶을 때**  
  → 후처럼 “자료로 학습 → 문서로 정리 → 우선순위 정해서 적용”이 유리함.

- **실무 추천**  
  - 1단계: 연동/일관성 문제는 **전** 방식으로 빠르게 수정  
  - 2단계: 급여·계약·설정 등은 **후** 방식으로 학습 문서 만들고, 그중에서 “선지급”, “공제 상세” 등부터 골라서 적용  

이렇게 하면 **전**의 빠른 수정과 **후**의 방향성·문서화 장점을 같이 쓸 수 있습니다.
