# 인사관리 관리자 전용 기능 이동 보고서

작성일: 2026-04-01  
대상 범위: 인력관리, 근태/연차, 급여, 복지/문서, 계약 관련 인사관리 기능

## 1. 결론 요약

현재 인사관리 권한은 `hr_근태`, `hr_연차휴가`, `hr_급여`, `hr_문서보관함`처럼 영역 단위로만 열려 있어, 일반 인사담당자가 회사 전체 정책을 바꾸는 기능까지 함께 보게 됩니다. 특히 아래 항목은 악용 여지가 높거나 실수 한 번으로 전사 데이터가 바뀌는 기능이라 관리자 전용으로 분리하는 것이 맞습니다.

우선 이동 대상 1순위:

- 연차사용촉진 자동화
- 연차 자동부여 설정
- 공휴일 달력 자동 반영
- 휴일/대체휴무 규칙
- 인사통합설정의 근태·급여 정책 항목
- 급여 통합설정 / 세율·보험요율 / 급여월 마감 잠금 / 급여고도화센터
- 근무표 자동편성의 규칙 저장, 패턴 저장, 월간 저장본 덮어쓰기
- 문서보관함의 전사 문서 생성·수정·삭제
- 계약관리의 비과세 항목 설정, 연봉 계약 갱신 설정

반대로 직원 단위 운영 업무는 그대로 인사담당자에게 남겨도 됩니다.

- 근태 조회 / 휴가 신청내역 확인
- 직원 기본정보 수정
- 증명서 발급 이력 기록
- 경조사 등록
- 계약 발송 및 서명 진행 추적
- 직원 단위 보험/복지 체크

## 2. 현재 권한 구조 문제

### 2-1. 탭 접근이 너무 넓게 열려 있음

- `app/main/기능부품/인사관리.tsx:293-297`
  - `연차/휴가` 탭은 `hr_연차휴가` 또는 `hr_근태`만 있어도 접근됩니다.
- `app/main/기능부품/인사관리.tsx:813-822`
  - `LeaveManagement` 내부에 정책성 탭이 섞여 있고, 일반 인사 권한으로 같이 들어갑니다.
- `lib/feature-permissions.ts:58-68`
  - 인사 권한이 영역 단위만 있고, “정책 수정”이나 “자동화 실행” 같은 세부 권한이 없습니다.
- `lib/access-control.ts:436-440`
  - 인사 상세 권한도 결국 큰 묶음 권한으로 평가되어, 민감 기능을 별도로 차단하지 못합니다.

### 2-2. 운영 업무와 시스템 설정이 같은 화면에 섞여 있음

다음 화면들은 “조회/처리”와 “회사 정책 변경”이 같은 화면에 공존합니다.

- 휴가관리메인
- 급여관리
- 계약관리
- 문서보관함
- 근무표자동편성

이 구조에서는 권한이 조금만 넓어져도 정책 변경 버튼이 그대로 노출됩니다.

## 4. 영역별 상세 판단

### 4-1. 근태 / 연차

`휴가관리메인`에는 신청내역 조회와 함께 정책성 탭이 같이 있습니다.

- 탭 정의: `app/main/기능부품/인사관리서브/휴가신청/휴가관리메인.tsx:29-46`
- 자동부여 실행: `app/main/기능부품/인사관리서브/휴가신청/휴가관리메인.tsx:184-199`
- 전사 알림 발송: `app/main/기능부품/인사관리서브/휴가신청/연차촉진시스템.tsx:110`
- 정책 저장: `app/main/기능부품/인사관리서브/휴가신청/휴일근무규칙설정.tsx:68`, `lib/leave-policy-settings.ts:140`

판단:

- `연차/휴가 신청내역`, 직원별 사용 현황 조회는 인사담당자 유지
- `연차사용촉진 자동화`, `연차 자동부여 설정`, `공휴일 달력`, `휴일/대체휴무 규칙`은 관리자 전용 이동

### 4-2. 인력관리 / 근무표

`근무표자동편성`은 자동 생성 자체보다 “규칙을 저장하고 팀 운영 기준을 굳히는 기능”이 더 위험합니다.

- 현재 관리자 판정: `app/main/기능부품/근무표자동편성.tsx:2460`
  - `role === 'admin'` 외에 `company === 'SY INC.'`, `permissions.mso === true`도 포함하고 있어 기준이 느슨합니다.
- 패턴/규칙 저장: `app/main/기능부품/근무표자동편성.tsx:4501-4622`
- 월간 저장본 덮어쓰기: `app/main/기능부품/근무표자동편성.tsx:5692-5735`

판단:

- 월간 초안 생성, 시뮬레이션, 검토는 팀장/인사운영까지 허용 가능
- 저장 규칙, 패턴 저장/삭제, 실제 월간 저장 반영은 관리자 또는 최소 부서 관리자 이상 별도 권한으로 축소 필요

### 4-3. 급여

현재 급여 화면은 운영 업무와 정책/거버넌스 기능이 한 화면에 섞여 있습니다.

- 통합설정 탭 노출: `app/main/기능부품/인사관리서브/급여관리.tsx:282`, `app/main/기능부품/인사관리서브/급여관리.tsx:416-417`
- 세율·보험요율 저장: `app/main/기능부품/인사관리서브/급여명세/세율보험요율관리.tsx:164-166`
- 급여월 잠금/해제: `app/main/기능부품/인사관리서브/급여명세/급여월마감잠금.tsx:98-154`
- 정책버전 및 고위험 처리: `app/main/기능부품/인사관리서브/급여명세/급여고도화센터.tsx:598-610`, `:659-792`, `:1023-1037`

판단:

- 급여 조회, 급여정산 실행, 명세 확인은 인사급여 담당자 유지
- 세율/보험요율, 월마감 잠금, 승인상태 저장, 정책 버전, 가압류/상계, 프리랜서 지급은 관리자 전용 이동

### 4-4. 복지 / 문서 / 계약

복지 쪽은 전사 정책 변경보다는 직원 단위 입력이 많아 모두 관리자 전용으로 돌릴 필요는 없습니다.

- 직원 단위 보험/복지 체크: `app/main/기능부품/인사관리서브/구성원현황.tsx:1486-1525`
  - 직원 개별 정보라 유지 가능
- 문서 원본/버전 수정·삭제: `app/main/기능부품/인사관리서브/문서보관함.tsx:173-198`, `:219-221`
  - 전사 문서 구조를 손댈 수 있어 관리자 권한이 맞음
- 계약관리 설정 블록: `app/main/기능부품/인사관리서브/계약관리.tsx:227-348`
  - 비과세 항목, 계약 갱신 기본값은 정책성 설정

판단:

- `구성원현황`의 직원별 보험/복지 체크는 유지
- `문서보관함`은 조회/다운로드는 유지, 생성/수정/삭제는 관리자 전용 또는 문서관리 전용 권한으로 분리
- `계약관리`는 발송/추적은 유지, 비과세·갱신설정은 관리자 전용 이동

## 5. 이동하지 않아도 되는 기능

아래 기능은 현재 범위에서는 관리자 전용으로 옮길 필요가 낮습니다.

- 직원 기본정보 조회 및 일반 수정
- 휴가 신청내역 조회 및 결재 연동 확인
- 근태기록 조회
- 경조사 신청/등록
- 증명서 발급 요청/이력 기록
- 계약 발송 및 대상자별 서명 상태 추적
- 직원 단위 사회보험/복지 체크

이 항목들은 “회사 전체 정책을 바꾸는 기능”이 아니라 “개별 건 처리” 성격이 강합니다.

## 6. 권한 설계 권고

현재처럼 `hr_근태`, `hr_연차휴가`, `hr_급여`만으로는 부족합니다. 아래처럼 세분화하는 것이 좋습니다.

- `admin_hr_leave_automation`
- `admin_hr_holiday_policy`

- `admin_payroll_policy`

권장 원칙:

- 조회 권한과 정책 변경 권한 분리
- 일괄 반영 버튼과 저장 버튼은 관리자 권한 이중 체크
- 화면 숨김만이 아니라 서버 저장 API에서도 권한 검증

## 8. 최종 권고안
